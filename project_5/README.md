# Лабораторна робота №5: Expo Router та Auth Guards
## 🎯 Мета
Ознайомлення з сучасною file-based маршрутизацією та реалізація захищених сторінок.
## 🛠 Що було використано
- **Навігація:** `expo-router` (групи `(app)`, `(auth)`, динамічні рути `[id]`).
- **Стан:** Глобальний `AuthContext` для імітації сесії.
## 📱 Опис застосунку
Застосунок розділено на публічну зону (вхід/реєстрація) та захищену (каталог товарів). Завдяки Expo Router створено механізм редиректу: якщо користувач не авторизований, його примусово "викидає" на сторінку логіну. Після входу доступний список продуктів із динамічними сторінками деталей.

## ⚙️ Як це працює: Expo Router та Auth Guards
У цьому проєкті реалізовано сучасний підхід до навігації за допомогою **Expo Router** (file-based routing) та захисту приватних маршрутів (Auth Guards).
### 1. Глобальний стан авторизації (Context API)
Уся логіка авторизації винесена в окремий контекст `AuthContext.tsx`. Він зберігає стан `isAuthenticated` та надає функції для входу (`login`), реєстрації (`register`) та виходу (`logout`). 
У файлі `app/_layout.tsx` кореневий компонент обгорнутий в `<AuthProvider>`, що робить стан авторизації доступним у будь-якій точці застосунку.
### 2. Розділення маршрутів на групи
Структура директорії `app` використовує логічні групи (папки з круглими дужками), які не впливають на URL, але дозволяють розділити публічні та приватні екрани:
* **`(auth)`** — публічна зона. Містить екрани входу (`login.tsx`) та реєстрації (`register.tsx`).
* **`(app)`** — захищена зона (каталог товарів та їх деталі). Доступ сюди дозволено лише авторизованим користувачам.
### 3. Механізм Auth Guard (Редиректи)
Головний механізм захисту реалізований у файлі `app/(app)/_layout.tsx`. Перед тим як відрендерити екрани каталогу, цей Layout перевіряє стан авторизації через хук `useAuth()`.
Якщо користувач не авторизований (`!isAuthenticated`), компонент повертає редірект на логін
